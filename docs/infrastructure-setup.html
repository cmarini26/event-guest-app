Phase A

EC2 Server Setup (one-time)

1. Launch EC2 instance

  • Go to EC2 → Launch Instance
  • AMI: Ubuntu Server 22.04 LTS
  • Instance type: t3.small
  • Key pair: create or select an existing SSH key pair — download the .pem file
  • Security group: allow inbound 22 (SSH), 80 (HTTP), 443 (HTTPS)
  • Storage: 20 GB gp3
  • Launch the instance

2. Allocate and attach an Elastic IP

  • EC2 → Elastic IPs → Allocate Elastic IP address
  • Associate it with your new instance
  • Record this IP — every project's DNS A record points here

3. Install the server stack

SSH in: ssh -i your-key.pem ubuntu@<elastic-ip>

sudo apt update && sudo apt upgrade -y

# PHP 8.3
sudo add-apt-repository ppa:ondrej/php -y
sudo apt install -y php8.3-fpm php8.3-cli php8.3-pgsql php8.3-mbstring \
  php8.3-xml php8.3-curl php8.3-zip php8.3-bcmath php8.3-intl

# Nginx, Node, Certbot
sudo apt install -y nginx nodejs npm certbot python3-certbot-nginx unzip git

# Composer
curl -sS https://getcomposer.org/installer | php
sudo mv composer.phar /usr/local/bin/composer

4. Create a deploy user

GitHub Actions will SSH as this user. Never use root or ubuntu for deployments.

sudo useradd -m -s /bin/bash deploy
sudo mkdir -p /home/deploy/.ssh
sudo touch /home/deploy/.ssh/authorized_keys
sudo chmod 700 /home/deploy/.ssh
sudo chmod 600 /home/deploy/.ssh/authorized_keys
sudo chown -R deploy:deploy /home/deploy/.ssh
sudo usermod -aG www-data deploy

# Paste GitHub Actions public key into authorized_keys:
# sudo nano /home/deploy/.ssh/authorized_keys

Give deploy passwordless permission to restart Supervisor workers:

sudo visudo
# Add this line:
deploy ALL=(ALL) NOPASSWD: /usr/bin/supervisorctl
Phase B

RDS PostgreSQL (one-time)

  • RDS → Create database → Standard create → PostgreSQL 16
  • Template: Free tier (or Dev/Test for Multi-AZ later)
  • Instance class: db.t3.micro
  • Storage: 20 GB gp3, enable auto-scaling
  • Master username: postgres, generate a strong password
  • VPC: same VPC as EC2
  • Security group: create a new one, allow port 5432 from EC2's private IP only — never public
  • Disable public accessibility
  • Create the database, then note the endpoint hostname

Create a database per project

# From EC2, connect to RDS:
psql -h <rds-endpoint> -U postgres

# Then in psql:
CREATE DATABASE guestlist;
CREATE USER guestlist WITH PASSWORD 'strong-password-here';
GRANT ALL PRIVILEGES ON DATABASE guestlist TO guestlist;
\c guestlist
GRANT ALL ON SCHEMA public TO guestlist;
Phase C

S3 File Storage (one-time)

Create a bucket per project

  • S3 → Create bucket
  • Name: guestlist-production-files (must be globally unique)
  • Region: us-east-1 (same as EC2 for low latency)
  • Block all public access: ON (files served via signed URLs)
  • Versioning: off (not needed for uploads)

Create IAM user + policy

  • IAM → Users → Create user — name: guestlist-app
  • Attach a custom policy (not Admin — least privilege):
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject",
        "s3:PutObjectAcl"
      ],
      "Resource": "arn:aws:s3:::guestlist-production-files/*"
    },
    {
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::guestlist-production-files"
    }
  ]
}
  • Create Access Key (CLI/SDK type) → save AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY

CORS policy (required for direct browser uploads)

[
  {
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["GET", "PUT", "POST", "DELETE"],
    "AllowedOrigins": ["https://yourdomain.com"],
    "ExposeHeaders": ["ETag"]
  }
]
Phase D

Per-Project Server Configuration

Repeat these steps for each new project deployed to this server.

1. Create app directory

sudo mkdir -p /var/www/guestlist
sudo chown deploy:www-data /var/www/guestlist
sudo chmod 775 /var/www/guestlist

# Clone the repo as deploy user:
sudo -u deploy git clone git@github.com:cmarini26/event-guest-app.git /var/www/guestlist

2. Nginx virtual host

# /etc/nginx/sites-available/guestlist
server {
    listen 80;
    server_name yourdomain.com;
    root /var/www/guestlist/public;
    index index.php;

    client_max_body_size 20M;

    add_header X-Frame-Options "DENY";
    add_header X-Content-Type-Options "nosniff";

    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    location ~ \.php$ {
        fastcgi_pass unix:/var/run/php/php8.3-fpm.sock;
        fastcgi_param SCRIPT_FILENAME $realpath_root$fastcgi_script_name;
        include fastcgi_params;
    }

    location ~ /\.(?!well-known).* {
        deny all;
    }
}

# Enable + SSL:
sudo ln -s /etc/nginx/sites-available/guestlist /etc/nginx/sites-enabled/
sudo certbot --nginx -d yourdomain.com
sudo nginx -t && sudo systemctl reload nginx

4. First-time bootstrap

cd /var/www/guestlist
composer install --no-dev --optimize-autoloader
npm ci && npm run build
php artisan key:generate
php artisan migrate --force
php artisan optimize
# storage:link not needed — using S3

5. Queue worker (Supervisor)

# /etc/supervisor/conf.d/guestlist-worker.conf
[program:guestlist-worker]
command=php /var/www/guestlist/artisan queue:work database --tries=3 --sleep=3 --max-time=3600
user=deploy
numprocs=2
autostart=true
autorestart=true
redirect_stderr=true
stdout_logfile=/var/log/guestlist-worker.log
sudo supervisorctl reread && sudo supervisorctl update
sudo supervisorctl start guestlist-worker:*

6. Scheduler

# Run as deploy user:
crontab -e -u deploy
# Add:
* * * * * php /var/www/guestlist/artisan schedule:run >> /dev/null 2>&1
Phase E

GitHub Actions CI/CD

Generate deploy SSH key pair

ssh-keygen -t ed25519 -C "github-actions-deploy" -f ~/.ssh/deploy_key -N ""
# Add public key to server's deploy user:
cat ~/.ssh/deploy_key.pub | ssh ubuntu@<elastic-ip> \
  "sudo tee -a /home/deploy/.ssh/authorized_keys"

Add GitHub Secrets

In the repo: Settings → Secrets and variables → Actions → New repository secret

EC2_HOSTYour Elastic IP address
EC2_SSH_KEYContents of ~/.ssh/deploy_key (private key)

Workflow file

Create .github/workflows/deploy.yml in the repo:

name: Test & Deploy

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: shivammathur/setup-php@v2
        with:
          php-version: '8.3'
          extensions: pdo_sqlite, mbstring, xml, curl, zip, bcmath

      - name: Install PHP dependencies
        run: composer install --no-interaction --prefer-dist

      - name: Prepare environment
        run: |
          cp .env.example .env
          php artisan key:generate

      - name: Run tests
        run: php artisan test

  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Build frontend assets
        run: npm ci && npm run build

      - name: Upload built assets to EC2
        uses: appleboy/scp-action@v1
        with:
          host: ${{ secrets.EC2_HOST }}
          username: deploy
          key: ${{ secrets.EC2_SSH_KEY }}
          source: "public/build"
          target: "/var/www/guestlist/public"
          strip_components: 1

      - name: Deploy application
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.EC2_HOST }}
          username: deploy
          key: ${{ secrets.EC2_SSH_KEY }}
          script: |
            set -e
            cd /var/www/guestlist

            git pull origin main

            composer install --no-dev --optimize-autoloader --no-interaction

            php artisan migrate --force
            php artisan optimize

            sudo supervisorctl restart guestlist-worker:*

            echo "Deploy complete at $(date)"
How it works Every push to main runs the full test suite first. If any test fails, the deploy step is skipped — broken code never reaches production. Frontend assets are built in the CI runner (not on the server), transferred via SCP, then the server pulls the PHP source and runs migrations.
Phase F

Third-Party Services

Resend (email)

  • Sign up at resend.com → Add domain → copy DNS records (SPF, DKIM, DMARC) to your DNS provider
  • Wait for domain verification (usually <5 minutes)
  • Create API key → RESEND_API_KEY
  • Set MAIL_FROM_ADDRESS=hello@yourdomain.com

Stripe

  • Dashboard → API Keys → copy STRIPE_KEY (publishable) and STRIPE_SECRET
  • Create Event Pass product: $19.00 one-time
  • Create Pro product: $29/month + $290/year (two prices)
  • Create Business product: $79/month + $790/year (two prices)
  • Copy the 4 recurring price IDs into env vars
  • Webhooks → Add endpoint: https://yourdomain.com/api/webhooks/stripe
  • Events: checkout.session.completed, customer.subscription.created, customer.subscription.updated, customer.subscription.deleted
  • Copy signing secret → STRIPE_WEBHOOK_SECRET
  • Settings → Billing → Customer portal → Enable

Google OAuth

  • console.cloud.google.com → New project → APIs & Services → Credentials
  • Create OAuth 2.0 Client ID (Web application type)
  • Authorized redirect URIs: https://yourdomain.com/auth/google/callback
  • Copy Client ID → GOOGLE_CLIENT_ID and Client Secret → GOOGLE_CLIENT_SECRET
Phase G

Go-Live Smoke Test

  • Health check: GET https://yourdomain.com/api/health returns {"status":"ok"}
  • Registration: create account → receive verification email → verify → land on dashboard
  • Event creation: create event → publish → RSVP as guest → host receives email notification
  • File upload: attach a PDF to an event → confirm URL is an S3 URL
  • Stripe Event Pass: checkout → Stripe test card → webhook fires → event_pass_paid_at set
  • Stripe subscription: upgrade to Pro → webhook fires → users.plan = pro
  • Password reset: full flow end-to-end
  • Google OAuth: sign in with Google → lands on dashboard
  • Admin: php artisan admin:promote your@email.com → /admin accessible
  • Queue worker: sudo supervisorctl status shows workers RUNNING
  • Uptime monitoring: set up UptimeRobot or Better Uptime on /api/health
Future

When to Scale Up

EC2 CPU consistently >70%Upgrade to t3.medium ($33/mo) or add ALB + Auto Scaling Group
DB CPU >70% or IOPS maxedUpgrade to db.t3.small, add read replica, or enable RDS Proxy
Queue depth growingIncrease Supervisor numprocs or move to SQS
Need zero-downtime deploysAdd ALB, use blue/green deploy via AWS CodeDeploy
Global audience / latencyAdd CloudFront CDN in front of EC2 + S3
Mobile push at scaleMigrate from FCM Legacy HTTP to Firebase Admin SDK or AWS SNS
